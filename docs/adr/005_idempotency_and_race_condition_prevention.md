# ADR-005: Idempotency and Race Condition Prevention in Sync-Aware UX

## Status
* Proposed

## Context (背景と課題)
完全オフライン環境での動作を前提とした Sync-Aware UX（ADR-001）を採用した本システムにおいて、技術監査により以下の2つの重大な「信頼性のリスク（静かなる失敗の要因）」が発見された。

1. **UI操作の冪等性（Idempotency）欠如:**
   現場スタッフがタブレットの「保存（Save）」ボタンを連続でタップ（連打）した場合、保存処理中にUIをロックする機構が存在しないため、同一の推論結果が IndexedDB (Dexie) に重複して大量保存され、データベースが汚染される。
2. **バックグラウンド同期の競合状態（Race Condition）:**
   保存完了後に自動発火する `syncRecords()` 処理に排他制御が実装されていない。通信環境が不安定（Flaky Network）な状態で複数回の保存が行われると、過去の同期プロセスが終わる前に新しいプロセスが走り、同一の未同期（pending）レコードをGCPへ二重に送信しようとするレースコンディションが発生する。

データセンターの監査エビデンスという性質上、データの重複や不整合はシステムの信頼性を根底から揺るがすため、厳格なガードレールが必要である。

## Decision (決定事項)
これらの問題を解決し、システムの堅牢性を担保するため、以下のアーキテクチャルールを導入する。

1. **UIレベルの排他制御（操作の冪等性担保）:**
   データ書き込みを伴うユーザー操作（保存ボタン等）に対しては、コンポーネントレベルで処理中の状態（`isSaving` 等）を管理し、実行中はボタンを直ちに非活性（`disabled`）にする等の排他制御を必須とする。
2. **サービス層における同期プロセスのミューテックス（競合排除）:**
   バックグラウンド同期を司る層（`syncService.ts` 等）には、現在同期中であるかを示す内部ステート（ミューテックス）を持たせる。すでに同期プロセスが実行中の場合、新たな同期要求が発火しても処理を安全にスキップ（早期リターン）する「ロック機構」を必ず実装する。

## Consequences (結果・影響)
* **[Good] データ整合性の完全保証:** ボタン連打や通信遅延による重複レコードの作成・二重送信を物理的に防ぎ、クラウド（GCP）に保存される監査証跡の正確性が担保される。
* **[Good] システムリソース・APIコールの最適化:** 不要な重複処理を間引くことで、端末側の負荷やクラウド側のAPIコール数（コスト）を抑制できる。
* **[Bad] 実装・テストの複雑化:** コンポーネントおよびサービス層での状態管理（ローディングフラグやミューテックス）が増えるため、コードベースがやや複雑になり、非同期競合をエミュレートするテストの記述難易度が上がる。
